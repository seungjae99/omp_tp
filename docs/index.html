ROS 2 / Ignition Gazebo

omp_tp

Unmanned Surface Vehicle simulation in Ignition Gazebo with camera-based obstacle detection and a Finite State Machine avoidance controller, bridged to ROS 2 Humble.

ROS 2 Humble Ignition Gazebo Python OpenCV FSM ament_python

Overview

omp_tp simulates a small Unmanned Surface Vehicle (USV) floating in a water environment. The boat is modeled with realistic buoyancy and hydrodynamics physics, controlled by two independent thrusters. A front-facing camera feeds a ROS 2 node that classifies whether an obstacle is present and drives a timed avoidance maneuver.

The project tracks the iterative development of the avoidance logic across three versions of the controller node, from a hardcoded state machine to a clean, data-driven design.

Three processes run in parallel from a single launch file:

  • Ignition Gazebo — physics simulation with buoyancy, hydrodynamics, thruster, and camera plugins active in the SDF world
  • ros_gz_bridge (parameter_bridge) — translates Ignition transport topics to ROS 2 messages and back
  • ObstacleAvoidNode — subscribes to the camera image, detects obstacles, and publishes thrust commands
  • RViz2 — displays the processed grayscale camera stream for monitoring

Obstacle Detection

Detection runs in the image callback at camera frame rate using OpenCV:

  • Convert BGR frame to grayscale
  • Threshold: pixels below 50 count as "dark"
  • If dark pixel count exceeds 10,000 the obstacle flag is set
  • Processed grayscale image is re-published to /gray_image

The threshold approach is intentionally simple: static obstacles in the simulation appear as large dark regions against the bright water surface, making pixel counting sufficient.

USV Model (SDF)

The boat is assembled from multiple links in a single SDF model:

  • Base hull (1.0 m x 0.5 m) with mass 5 kg
  • Left and right side walls (structural)
  • Two thruster plugins (differential drive)
  • Front-facing camera sensor
  • Graded buoyancy: water density 1000 kg/m3 above surface depth 0
  • Hydrodynamics plugin for damping forces

Controller Design: Three Iterations

The obstacle avoidance logic was refactored twice, progressively reducing coupling and improving maintainability:

v1 obs_avoid_1.py
8-state hardcoded FSM. Each state (AVOID_LEFT1, AVOID_FWD1, ... AVOID_LEFT2) is an explicit branch in the control loop. Thrust values and durations are plain instance variables. The logic is self-contained but difficult to modify.
v2 obs_avoid_2.py
Same 8-state FSM, cleaner structure. Constants extracted to class-level attributes. Repeated thrust calls replaced with _drive_forward(), _turn_left(), _turn_right() helpers. State transitions centralised in _transition().
v3 obs_avoid_3.py
Data-driven, 2-state design (final). The eight avoidance steps are encoded as a list of (action, duration) tuples. The FSM has only two states — NAVIGATING and AVOIDING — and a step index walks the sequence. Adding or reordering maneuvers requires only editing the list.

Avoidance Sequence (v3)

When an obstacle is detected, the USV executes a fixed S-curve bypass maneuver and returns to the original heading:

1
Turn Left
3.0 s — swing away from the obstacle
2
Forward
7.0 s — clear the obstacle laterally
3
Turn Right
3.0 s — realign parallel to original track
4
Forward
4.0 s — pass alongside the obstacle
5
Turn Right
3.0 s — angle back toward original track
6
Forward
7.0 s — travel back to original lane
7
Turn Left
3.0 s — restore original heading

After step 7 the FSM returns to NAVIGATING and the USV resumes forward motion.

Quick Start

# 1. Clone and build
mkdir -p ~/omp_tp/src
cd ~/omp_tp/src
git clone https://github.com/seungjae99/omp_tp.git
cd ~/omp_tp && colcon build
source install/setup.bash

# 2. Launch the Gazebo world (run from tp_worlds/)
cd ~/omp_tp/src/omp_tp/tp_worlds
ign gazebo tp_world.sdf

# 3. Play the simulation, then in a new terminal:
ros2 launch omp_tp omp_tp.launch.py

The launch file starts RViz2, the obstacle avoidance node (obs_avoid_3), and the ros_gz_bridge in a single command.
